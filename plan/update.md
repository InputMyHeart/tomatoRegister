# 番茄记账优化与维护计划

本文把当前项目的优化工作拆成可独立验收的步骤。默认原则是：每一步保持小程序可运行、可回退；先建立规范和边界，再进行结构调整与性能优化。

## 执行顺序

```text
Step 0 基线与备份
  -> Step 1 工程规范
  -> Step 2 公共基础层
  -> Step 3 前端页面与组件整理
  -> Step 4 云函数模块化
  -> Step 5 数据库与性能
  -> Step 6 测试、监控与发布
```

## Step 0：建立改造基线

**目标：** 明确当前可运行版本、数据结构和风险范围，保证后续每步可以验证和回退。

**工作内容：**

- 整理 `README.md`：本地启动、微信开发者工具配置、云环境、云函数部署步骤。
- 盘点云数据库集合：`users`、`ledgers`、`records`、`categories`、`ledgerInvites`，记录字段、权限和索引现状。
- 导出测试环境数据或建立可重复生成的测试数据脚本。
- 建立核心业务的人工回归清单：登录、建账本、记账、改/删记录、预算、邀请加入、分类管理。
- 将当前功能性修改提交为一个可回退的基线版本。

**验收标准：** 新成员可按文档运行项目；核心流程有明确的回归检查项；数据结构有文档可查。

## Step 1：统一工程规范

**目标：** 让代码风格、命名和文件编码稳定一致，减少无意义的格式差异。

**工作内容：**

- 新增前端工程配置：`package.json`、ESLint、Prettier、`.editorconfig`。
- 统一 JavaScript 格式：2 空格缩进、统一引号和分号策略、禁止压缩成单行的页面逻辑。
- 统一 UTF-8 编码和换行策略，避免不同终端读取中文时出现乱码。
- 制定命名约定：目录/路由用 `kebab-case`，变量与函数用 `camelCase`，常量用 `UPPER_SNAKE_CASE`。
- 统一页面、组件、云函数的文件结构；历史命名直接迁移，并同步修改全部页面路径和调用方。
- 约定提交信息格式，如 `feat:`、`fix:`、`refactor:`、`docs:`。

**验收标准：** `lint` 和 `format:check` 可执行；全项目格式稳定；新增代码遵循同一套命名规则。

## Step 2：抽取前端公共基础层

**目标：** 清除页面中重复的云函数调用、鉴权、格式化和数据映射逻辑。

**工作内容：**

- 新建 `miniprogram/services/api.js`：统一封装 `wx.cloud.callFunction`、响应解包、错误转换、请求日志。
- 新建领域服务：`ledger.service.js`、`record.service.js`、`category.service.js`、`profile.service.js`。
- 新建 `utils/money.js`：金额解析、两位小数处理、展示格式。
- 新建 `utils/date.js`：日期解析、月份范围、展示格式。
- 新建 `utils/auth.js`：登录态恢复、页面鉴权、登录页跳转。
- 新建 `utils/mapper.js`：统一处理 `id/_id`、账本角色、头像默认值等前端展示映射。
- 先迁移首页、明细页、记录详情页中重复的金额/记录映射逻辑，再迁移其他页面。

**验收标准：** 页面不再直接散落调用 `wx.cloud.callFunction`；重复的金额、日期和账本映射只保留一份实现。

## Step 3：整理页面和组件边界

**目标：** 降低页面复杂度，让可复用 UI 和业务状态各自归位。

**工作内容：**

- 保持页面只负责路由参数、生命周期和页面级状态；将展示区块拆为业务组件。
- 优先抽取：`record-form`、`category-tree`、`ledger-selector`、`member-list`、`budget-progress`。
- 给自定义组件统一前缀，例如 `tl-`，并定义组件 API：properties、events、插槽和默认状态。
- 建立全局设计令牌：颜色、间距、圆角、字号、阴影、z-index，减少页面内硬编码样式。
- 精简 `ri-icon`：根据实际使用图标生成子集或改为精简映射，降低小程序包体。
- 逐步统一页面 `index.json`，删除重复的组件声明，仅保留页面特有配置。

**验收标准：** 首页、记录编辑、分类管理等复杂页面的业务 UI 被拆成可独立复用的组件；样式常量不再在多页复制。

## Step 4：模块化云函数

**目标：** 将单文件云函数拆成清晰的业务模块，并同步迁移全部调用方，不保留旧 `action` 接口。

**建议目录：**

```text
cloudfunctions/tomatoLedger/
  index.js                 # action 分发与统一异常处理
  actions/
    auth.js
    ledger.js
    record.js
    category.js
    invite.js
  repositories/
    user.repository.js
    ledger.repository.js
    record.repository.js
    category.repository.js
  policies/
    ledger-access.js
  validators/
    schemas.js
  shared/
    response.js
    errors.js
    constants.js
```

**工作内容：**

- 将 `ok/fail`、错误码、参数校验、日志收敛到 `shared`。
- 将用户、账本、记录、分类、邀请业务按 action 拆分。
- 将数据库查询/写入抽到 repository，避免业务层直接散落 `db.collection(...)`。
- 将账本角色判断、只读限制、所有者限制集中到 `policies/ledger-access.js`。
- 对所有写操作做参数校验，并且始终从服务端重新判断用户与账本权限。
- 对邀请加入、删除账本、分类删除并迁移记录等多步写操作使用事务或幂等策略。
- 完成全部调用方迁移后移除旧 action 和旧返回结构；需要 API 版本化时采用明确的新版本接口。

**验收标准：** `index.js` 只负责初始化、路由和异常处理；每个业务模块可以独立阅读与测试；原有页面调用不需要同步大改。

## Step 5：数据库、性能与数据演进

**目标：** 确保数据增长后仍能保持正确性和响应速度。

**工作内容：**

- 记录列表与统计接口改用游标分页，不再依赖固定 `limit(100)`。
- 设计并建立索引：记录的账本/日期组合、邀请码 token、账本邀请码和只读邀请码等。
- 明确统计计算策略：实时计算、按月缓存或增量聚合；优先保证金额正确，再评估缓存。
- 为金额、日期、账本和分类实体定义字段约束与默认值。
- 增加 `schemaVersion` 和数据迁移策略，处理将来的字段调整。
- 设计账本删除、成员退出、邀请过期、分类删除迁移的边界规则和恢复策略。
- 为生产环境建立备份/导出流程。

**验收标准：** 明细可分页浏览；核心查询有索引说明；多步数据修改具备一致性保护；数据模型可安全演进。

## Step 6：测试、监控与发布流程

**目标：** 让改动可验证、问题可定位、发布可追溯。

**工作内容：**

- 为纯函数添加单元测试：金额、日期、参数校验、权限规则、分类迁移逻辑。
- 为云函数 action 添加集成测试：登录、建账本、记录 CRUD、邀请、分类管理。
- 建立回归测试清单，并在每次发布前执行。
- 统一云函数日志字段：`action`、用户标识、账本 ID、耗时、错误码、请求 ID。
- 建立错误分级和用户提示策略：用户可理解的提示与开发排错日志分离。
- 区分开发、测试、生产云环境；不把生产环境标识直接硬编码在业务代码中。
- 增加 CI：格式检查、Lint、测试、构建/上传前校验。

**验收标准：** 合并前能自动完成基础质量检查；线上错误可按 action 和请求定位；发布流程可重复执行。

## 建议的实施节奏

1. 先完成 Step 0 和 Step 1，建立可回退、可检查的改造基础。
2. 接着完成 Step 2，减少前端重复代码后再动页面结构。
3. Step 3 与 Step 4 可分批推进：先拆最常改的记录和分类域。
4. Step 5、Step 6 作为发布前的可靠性门槛，不能只在功能完成后补做。

## 每次改动的通用要求

- 单次提交只处理一个明确目标。
- 修改前确认页面路径、组件引用、云函数 action 和数据库字段影响范围。
- 修改后至少执行格式校验、静态检查和相关人工回归。
- 涉及数据删除、迁移、权限规则和生产环境时，先备份并给出回滚方案。
